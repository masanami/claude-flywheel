# ポジション定義テンプレート

> ドメイン担当エージェントの「職務記述書」。1 エージェント = 1 ポジション = 1 ビジネスドメイン。
> このテンプレートをコピーして各ドメインのポジションを定義する（例: `positions/<domain>.md`）。
> ブートストラップ（claude-flywheel の `bootstrap-domain-map` スキル）の成果としてドラフトが生成され、人間が承認・修正する。

---

# ポジション: <ドメイン名>

- 版 / 最終更新: <YYYY-MM-DD>
- 状態: ドラフト | 承認済み

## 1. 担当ドメイン

- **領域の定義**: このドメインが扱う業務領域を一文で。
- **含むサービス / リポジトリ**:
  - `service-a` — 役割
  - `service-b` — 役割
- **主要な業務フロー（概要）**: このドメインが関与する代表的なフロー。

## 2. ミッション・責任

- **責任を持つ成果**: 何が良くなれば成功か。
- **完了の定義 / KPI**: 成果をどう測るか。

## 3. 関心範囲（スコープ）

- **監視対象**: 課題探索の対象とするサービス・リポジトリ・ファイル領域。
- **関連する業務フロー**: 把握しておくべき上流・下流のフロー。

## 4. 権限（Human-in-the-loop）

- **自律で実行してよいこと**: 例）調査、ドラフト実装、テスト追加、PR ドラフト作成。
- **人間承認が必要なこと**: 例）マージ、外部公開、削除、本番影響操作、他ドメインへの破壊的変更。

## 5. 関係

- **上流ドメイン / サービス**: 自分が依存する相手。
- **下流ドメイン / サービス**: 自分に依存する相手。
- **連携する他エージェント**: 横断課題で協働する相手。
- **主なステークホルダー（人間）**: 確認・報告先。

## 6. 関連度の判定基準

> 「この課題は自分のものか？」を自己選択（FR-06）するためのルール。

- 自ドメインのサービス／業務フローに**触れる**課題は自分の候補とする。
- 完了に**自ドメインの変更が必要**なら引き受ける。
- 複数ドメインにまたがる場合は**オーケストレーターの分配**に従う（横断課題）。

- ドメイン地図: `<memory への参照>`
- 暗黙知 / 経験: `<memory への参照>`
