# bootstrap-domain-map

横断的なマイクロサービス群を探索し、**ドメイン地図**・**ポジション案**・**エージェントmemoryの初期seed**を生成するブートストラップスキル。

> 出発点は「人間が業務フロー・サービス関係を把握しきれていない」状態。最初から綺麗なドメイン定義は無い前提で、地図づくりから始める。

## 前提・入力

- 対象は **このエージェントの守備範囲**（ドメイン）。既定は 1 ポジション、広い守備範囲ならサブ領域で複数ポジションに分ける。
- 対象とするリポジトリ群（モノレポ or 複数リポジトリ）のパスまたは URL。
- 不明な場合は、まず探索対象の範囲を人間に確認する。

## 手順

### 1. インベントリ（サービス一覧）
- サービス定義を収集する: マニフェスト（`docker-compose.yml`, k8s, `*.proto`, OpenAPI, `package.json` 等）、デプロイ設定、ディレクトリ構成。
- 各サービスの「名前・役割・技術スタック・エントリポイント」を抽出。

### 2. 関係抽出（サービス間の依存）
- サービス間の連携を解析: HTTP/RPC 呼び出し、イベント/メッセージ（キュー・トピック）、共有 DB/テーブル、API クライアント設定。
- 「誰が誰を呼ぶ / 誰のイベントを購読するか」の有向グラフを作る。

### 3. 業務フロー推定
- 主要なエントリポイント（API・UI・ジョブ）から代表的なユースケースを辿り、業務フローを推定する。
- コメント・README・ADR・命名から**暗黙知の候補**（業務ルール・制約・注意点）を拾う。

### 4. ドメインクラスタリング
- 関係（呼び出し頻度・データ結合・業務フロー）が強いサービス群を**ドメイン**に束ねる。
- ドメイン境界と、ドメイン間の依存（上流/下流）を定義する。

### 5. 成果物の生成

> **出力規約（AI 可読性）**: 生成物（ドメイン地図・ポジション案・memory）の図は **mermaid**（ASCII art 不可）、**各図に1行キャプション**、**全コードフェンスに言語タグ**（ディレクトリツリーは `text` 例外）。

- **ドメイン地図**: サービス・関係・ドメイン境界（**mermaid 図** + 表）。memory `map` として出力。
- **ポジション案**: `${CLAUDE_PLUGIN_ROOT}/templates/position.md` を読み込み、各項目を埋めたドラフトを `positions/<domain>.md` に生成。
- **関連リポジトリのマニフェスト**: 手順1〜2で把握した対象リポジトリを `repos.tsv` に列挙する（形式は `${CLAUDE_PLUGIN_ROOT}/templates/repos.tsv`：`<name> <url> <branch>`、空白/タブ区切り・`#` コメント）。以後 `${CLAUDE_PLUGIN_ROOT}/scripts/sync-repos.sh` が `.flywheel/repos/`（作業用）へ clone/fetch する。**秘密情報（認証）は書かない**。memory `map` から該当リポジトリを `repos.tsv` の `<name>` で参照できるようにする。
- **課題の取り込み元候補（任意）**: 手順1〜4 で把握したドメイン・対象 repo・関連ドキュメントから、**このドメインの課題がどこに書かれるか**（担当 repo の Issue/ファイル、関連する Notion/Doc/Slack 等）を推定し、`challenge-sources.md` に**ソース候補をドラフト**する（雛形は `${CLAUDE_PLUGIN_ROOT}/templates/challenge-sources.md`、宣言項目は `id`/`type`/`locator`/`filter`。取り込みは [ingest-challenges](../ingest-challenges/SKILL.md) が担う）。`repos.tsv` を起こすのと同じ材料からの下書きで、**確証は低い前提**（`type`・`filter` は候補として提示し、step 6 で人間が確定）。**秘密情報（トークン・資格情報）は書かない**（認証は実行者環境に委ねる）。外部ソースを使わない・不明な場合はスキップ（初期は内部台帳直接記入だけでも運用可）。
- **memory seed**: **agent-memory スキルの save** で記録する（種別: `map`=サービス/関係/境界、`reference`=API契約・スキーマ等の所在、`tacit`=暗黙知候補）。frontmatter・INDEX・重複排除は agent-memory に委譲。**暗黙知などの推定は `confidence: low`**。

### 6. 人間レビュー（承認ポイント）
- ドメイン境界・ポジション案・暗黙知候補、および（生成していれば）課題の取り込み元候補（`challenge-sources.md`）を人間に提示し、承認・修正を得る。
- 確認済みの暗黙知は `confidence` を引き上げる。
- 推測が混じる箇所・カバーできなかった範囲は**明示**する（沈黙の切り捨てをしない）。

## 注意

- コードは読み取りのみ。対象リポジトリを変更しない。
- 秘密情報（認証情報・接続文字列）を memory やログに残さない。
