# run-cycle

Flywheel の 1 周（観測 → 整理 → 計画 → 実行 → 検証 → 学習 → 報告）を実行する「サイクル本体」。自律実行ランタイム（routine/cron）から定期起動される想定。

> 状態は課題台帳・記憶・ポジション（すべてファイル）。設計の詳細は claude-flywheel の `docs/architecture.md`（§5.2/§7/§3.8）を参照（実行には不要）。
> **ツール非依存**: 実作業は各ポジションが指定する接続ツールへ委譲する（開発例: claude-harness）。本スキルは特定ツールに依存しない。

## 前提

- ブートストラップ済み（`positions/`・`memory/` が存在。未整備なら bootstrap-domain-map スキルを先に実行）。
- 利用先ワークスペースに `challenge-ledger.md`（課題台帳）が存在する（未整備なら flywheel-init スキルで生成）。

## 入力（任意）

- 対象ドメイン（省略時は全ドメイン）。
- `--dry-run`: 変更・委譲・コミットを行わず、「やること」だけ報告する。

## 冪等性の原則（cron で安全に回すため）

- 各課題の**ステータスに基づき、着手可能なものだけ**処理する。
- 何度起動しても**二重実行しない**（ステータスで判定）。
- 人間承認ゲートに達した項目は**「提案を残して保留」し、サイクルは止めない**（スケジュール実行では人間をインラインで待たない）。
- 1 周で扱う件数に上限を設けてよい（残りは次サイクル）。

## 手順

### 0. 観測・取り込み（ingestion）
- 共有課題ソースから **自分（このエージェント）に関係する課題だけ**を自リポジトリの `challenge-ledger.md` へ取り込む。取り込み済みの課題は重複登録しない（冪等）。
- ベースライン（ポジション要約・記憶 INDEX・recall 手順）は `CLAUDE.md` として**自動ロード済み**。加えて `challenge-ledger.md` をロードし、**詳細記憶は手順2の recall で関連分だけ**引く。
- **（任意）関連リポジトリの最新化**: この周で関連リポジトリを参照／実作業する場合のみ、`${CLAUDE_PLUGIN_ROOT}/scripts/sync-repos.sh` を実行して `repos.tsv` のクローンを `.flywheel/repos/` に最新化する（冪等。作業用クローン＝編集・ブランチ・コミット可。同期は安全側で、dirty/別ブランチは更新せずローカル作業を保持。`repos.tsv` が無ければスキップ）。

### 1. 整理（自ポジションへの分類）── 対象: ステータス「未分類」
- 各課題を、自分のどのポジションが引き受けるか分類して「分類欄」を更新（形式は `${CLAUDE_PLUGIN_ROOT}/docs/challenge-ledger-format.md`）:
  - **担当ポジションが明確** → そのポジションを割当、ステータス → 分類済（単一ポジションのエージェントでは自明）。
  - **守備範囲か曖昧／他エージェントと重複しそう** → **要相談**（人間へ。承認ゲート #2）。
- ※ エージェント間の横断・振り分けは本スキルの範囲外（fleet スコープ＝人間）。

### 2. 計画 ── 対象: ステータス「分類済」で自分が引き受けたもの（自己選択）
- 関連記憶を前ロードする（**agent-memory スキルの recall**）。
- 目標 → タスクに分解し、タスク案を課題へ追記する。
- **【承認ゲート FR-13】** タスク起票は人間承認。「要承認」を付与して保留し報告。承認済みのみ次へ進む。

### 3. 実行 ── 対象: 承認済みタスク（ステータス → 着手中）
- ポジションが指定する**接続ツール**を使って実作業する。
- **作業対象（cwd）は作業用クローン `.flywheel/repos/<name>`**（`repos.tsv` の `<name>`、§3.9.1）。クローンが無ければ step 0 の `sync-repos.sh` で用意する。
- **委譲は cwd＝作業用クローンの独立 `claude -p` セッション**で接続ツール（開発例: claude-harness）を回す（設計の根拠は §3.9.2）。子セッションは対象 repo の `CLAUDE.md`＋`.claude/settings.json` をネイティブにロードするため、対象本来の設定で開発できる。多ターンは `-c`（`--continue`）/ `--resume <session-id>`（`-p` を先頭に置く）、非対話の権限は `--permission-mode` と **cwd の対象 repo の `.claude/settings.json`** に委ねる。
  - **【権限前提】自走委譲の spawn は事前許可が要る**: 親（エージェントrepo）から headless `claude -p` を spawn する行為は、事前許可が無いと Claude Code の auto-mode セーフティ分類器にブロックされ、routine/cron で実装ステップに到達できない。親ワークスペースの `.claude/settings.json` に `Bash(claude -p:*)` を **allow**（flywheel-init が scaffold）しておくこと。
  - **子に無制限 Bash を渡さない**: `--allowedTools Bash` のような“広範 Bash”は分類器の警戒対象。子の権限は **cwd の対象 repo の `.claude/settings.json`（allow/ask/deny）に統治させる**（設計どおりの委譲。§3.9.2）。
  - **不可（反例）**: メインセッションで直接 skill を回す（対象 `CLAUDE.md` はオンデマンド・`settings.json` 不適用）／サブエージェントに委譲（親設定のまま切り替わらない）。注入/union 方式は多 repo でスケールしないため不採用。
- クローン内の**ローカル編集・ブランチ作成・コミットは承認不要**（作業用クローンに閉じるため）。**push / PR 作成 / マージは破壊的・不可逆操作**として下記 FR-22 の承認ゲート対象。
- 手順2で recall した前提知識（`map` / `tacit` / `reference`）と**完了条件はブリーフ（指示）に明記して渡す**。独立 `claude -p` セッションは文脈を引き継がないため必須（暗黙知・関連サービス知識をレビュー/実装前に供給＝課題1ボトルネックBの解）。メインセッションが直接行うのは整理・計画など委譲前の作業に限る。
- **【承認ゲート FR-22】** 破壊的・不可逆操作（マージ／公開／削除／本番影響／**PR 作成（draft 含む）**）は**実行せず**、**内部の提案アーティファクト**（patch / handoff ノート＝作業用クローンのローカルブランチ＋コミット）を残して報告する。PR は人間承認後に作成する（line 53 と整合）。

### 4. 検証 ── 対象: 実装完了分（ステータス → 検証中）
- テスト／レビュー／動作確認を接続ツールで実施する。レビューをサブエージェント／別セッションで行う場合は、前提知識（`tacit` 等）を**レビュー観点としてブリーフに明記**し、暗黙知違反を検出させる。
- **【承認ゲート FR-32】** 最終確認は人間。検証結果を提示して保留する。

### 5. 学習（agent-memory スキルの save に委譲）
- `experience` を記録（効いた手順・失敗・レビュー指摘の傾向）。
- **good/bad の記録（自己改善ループ向け・append のみ／改修しない）**: その周の good / bad を `experience` に `outcome`・`target`・`signal`（・任意で `recurrence`）付きで残す。ここでは評価の集計や skill/サブエージェントの改修はしない。傾向の集計と改修提案は別スキル **reflect**（低頻度）が担う（設計は `${CLAUDE_PLUGIN_ROOT}/docs/self-improvement.md`）。
- 新たに判明した暗黙知を `tacit`（`confidence` 付き）で記録する。
- 課題のステータスを更新する。

### 6. 報告・コミット
- サイクルレポートを出力（処理件数 / 保留中の承認 / 要相談 / 次アクション）。
- 変更（台帳・記憶）を Git コミットする（`--dry-run` 時はスキップ）。
- **秘密情報をコミット・ログに残さない。**

## 出力

- サイクルレポート（人間向けサマリ）。
- 更新された `challenge-ledger.md` / `memory/`。

## 注意

- 接続ツールが無い領域では「調査・計画・記憶更新」までを行い、実作業は保留として報告する。
- 横断課題のファンアウト（複数ドメイン並列）は、必要に応じてワークフローの型で実装してよい。
- すべての段階で承認ゲート（本文中の **【承認ゲート】**）を維持する。
- **ハーネス自体（skill/サブエージェント/ブリーフ）の改修は本サイクルで行わない**。step 5 で good/bad を記録するだけにして、改修は別スキル reflect（低頻度）に委ねる（コストを抑えるための分離）。
