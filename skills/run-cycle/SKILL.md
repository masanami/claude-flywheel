# run-cycle

Flywheel の 1 周（観測 → 整理 → 計画 → 実行 → 検証 → 学習 → 報告）を実行する「サイクル本体」。自律実行ランタイム（routine/cron）から定期起動される想定。

> 状態は課題台帳・記憶・ポジション（すべてファイル）。設計の詳細は plugin リポジトリの `${CLAUDE_PLUGIN_ROOT}/docs/architecture.md`（§5.2/§7/§3.8）にあるが、**本スキルの実行には不要**（手順は本ファイルで自己完結）。
> **ツール非依存**: 実作業は各ポジションが指定する接続ツールへ委譲する（開発例: claude-harness）。本スキルは特定ツールに依存しない。

## 前提

- ブートストラップ済み（`positions/`・`memory/` が存在。未整備なら bootstrap-domain-map スキルを先に実行）。
- 利用先ワークスペースに `challenge-ledger.md`（課題台帳）が存在する（未整備なら flywheel-init スキルで生成）。

## 入力（任意）

- 対象ドメイン（省略時は全ドメイン）。
- `--dry-run`: 変更・委譲・コミットを行わず、「やること」だけ報告する。

## 冪等性の原則（cron で安全に回すため）

- 各課題の**ステータスに基づき、着手可能なものだけ**処理する。
- 何度起動しても**二重実行しない**（ステータスで判定）。
- 人間承認ゲートに達した項目は**「提案を残して保留」し、サイクルは止めない**（スケジュール実行では人間をインラインで待たない）。
- 1 周で扱う件数に上限を設けてよい（残りは次サイクル）。

## 手順

### 0. 観測・取り込み（ingestion）
- **取り込みは ingest-challenges スキルに委譲する**（`${CLAUDE_PLUGIN_ROOT}/skills/ingest-challenges/SKILL.md`）。共有課題ソースから **自分（このエージェント）に関係する課題だけ**を自リポジトリの `challenge-ledger.md` へ取り込む。取り込み元マーカーで照合し、取り込み済みの課題は重複登録しない（冪等。`fp` 変化時のみ人間記入欄を更新し分類・ステータスは保持）。取り込み元の宣言が無い（`challenge-sources.md` 不在）場合は、内部台帳の直接記入分だけで進める（このステップはスキップ可）。**このサイクル内で呼ぶ場合、コミットは step 6 でまとめて行うため ingest 側ではコミットしない**。
- ベースライン（ポジション要約・記憶 INDEX・recall 手順）は `CLAUDE.md` として**自動ロード済み**。加えて `challenge-ledger.md` をロードし、**詳細記憶は手順2の recall で関連分だけ**引く。
- **（任意）関連リポジトリの最新化**: この周で関連リポジトリを参照／実作業する場合のみ、`${CLAUDE_PLUGIN_ROOT}/scripts/sync-repos.sh` を実行して `repos.tsv` のクローンを `.flywheel/repos/` に最新化する（冪等。作業用クローン＝編集・ブランチ・コミット可。同期は安全側で、dirty/別ブランチは更新せずローカル作業を保持。`repos.tsv` が無ければスキップ）。

### 1. 整理（自ポジションへの分類）── 対象: ステータス「未分類」
- 各課題を、自分のどのポジションが引き受けるか分類して**「分類欄」を更新**する。分類欄はエージェントが記入する次の項目（人間記入欄＝説明・完了条件・緊急度は触らない）:
  - **担当ポジション**: `<position 名>`（単一ポジションのエージェントでは自明）。
  - **関連サービス**（任意）: `<service-a, service-b ...>`。
  - **優先度**: `P0 | P1 | P2`。
  - **ステータス**: `未分類 → 分類済 → 着手中 → 検証中 → 完了`。
  - **取り込み元**（外部取り込み分のみ・ingest-challenges が付与。手書き課題は空）。
  - **備考**（要相談の理由など）。
- 分類ルール:
  - **守備範囲に入り担当ポジションが明確**（記憶 `map` を参照） → そのポジションを割当、ステータス → 分類済。
  - **守備範囲か曖昧／他エージェントと重複しそう** → **要相談**（人間へ。承認ゲート #2）。備考に理由を書く。
  - **範囲外** → 取り込まない（共有ソースに残す）。
- ※ エージェント間の横断・振り分けは本スキルの範囲外（fleet スコープ＝人間）。
- （台帳フォーマットの詳細は plugin リポジトリの `${CLAUDE_PLUGIN_ROOT}/docs/challenge-ledger-format.md`。実行には上記で足りる。）

### 2. 計画 ── 対象: ステータス「分類済」で自分が引き受けたもの（自己選択）
- 関連記憶を前ロードする（**agent-memory スキルの recall**）。
- 目標 → タスクに分解し、タスク案を課題へ追記する。
- **【承認ゲート FR-13】** タスク起票は人間承認。「要承認」を付与して保留し報告。承認済みのみ次へ進む。

### 3. 実行 ── 対象: 承認済みタスク（ステータス → 着手中）
- ポジションが指定する**接続ツール**を使って実作業する。
- **作業対象（cwd）は作業用クローン `.flywheel/repos/<name>`**（`repos.tsv` の `<name>`）。クローンが無ければ step 0 の `sync-repos.sh` で用意する。
- **委譲は cwd＝作業用クローンの独立 `claude -p` セッション**で接続ツール（開発例: claude-harness）を回す。子セッションは対象 repo の `CLAUDE.md`＋`.claude/settings.json` をネイティブにロードするため、対象本来の設定で開発できる。多ターンは `-c`（`--continue`）/ `--resume <session-id>`（`-p` を先頭に置く）、非対話の権限は `--permission-mode` と **cwd の対象 repo の `.claude/settings.json`** に委ねる。
  - **【権限前提】自走委譲の spawn は事前許可が要る**: 親（エージェントrepo）から headless `claude -p` を spawn する行為は、事前許可が無いと Claude Code の auto-mode セーフティ分類器にブロックされ、routine/cron で実装ステップに到達できない。親ワークスペースの `.claude/settings.json` に `Bash(claude -p:*)` を **allow**（flywheel-init が scaffold）しておくこと。
  - **子に無制限 Bash を渡さない**: `--allowedTools Bash` のような“広範 Bash”は分類器の警戒対象。子の権限は **cwd の対象 repo の `.claude/settings.json`（allow/ask/deny）に統治させる**（設計どおりの委譲）。
  - **不可（反例）**: メインセッションで直接 skill を回す（対象 `CLAUDE.md` はオンデマンド・`settings.json` 不適用）／サブエージェントに委譲（親設定のまま切り替わらない）。注入/union 方式は多 repo でスケールしないため不採用。
- クローン内の**ローカル編集・ブランチ作成・コミットは承認不要**（作業用クローンに閉じるため）。**topic ブランチへの push・PR（draft 含む）作成は「外向きだが可逆」な操作としてサイクル内で自律実行してよい**（close / ブランチ削除で戻せる）。**真に破壊的・不可逆な操作（マージ／本番デプロイ・公開／削除／本番影響／force-push・`reset --hard` 等の履歴破壊）だけを**下記 FR-22 の承認ゲート対象とする（「外向き」と「不可逆」は別軸。区分の根拠は plugin の `${CLAUDE_PLUGIN_ROOT}/docs/architecture.md` §6・[#24](https://github.com/masanami/claude-flywheel/issues/24)）。
- 手順2で recall した前提知識（`map` / `tacit` / `reference`）と**完了条件はブリーフ（指示）に明記して渡す**。独立 `claude -p` セッションは文脈を引き継がないため必須（暗黙知・関連サービス知識をレビュー/実装前に供給＝課題1ボトルネックBの解）。メインセッションが直接行うのは整理・計画など委譲前の作業に限る。
- **【意思決定の主体】ブリーフに「あなたが意思決定者（ユーザー役）。停止せず決める」を必ず含める**: 委譲先には対話相手の人間がいない。接続ツールの**対話前提スキル**（例: claude-harness の `/define-feature`）が選択肢提示で人間の回答を待つと、意思決定で停止して先へ進めない。人間は意思決定権を委譲済みなので、委譲先エージェントは:
  - 要件・クリティカル設計の選択肢を**自分で評価し意思決定まで進める**（recall した `map`/`tacit`/`reference` とコード調査を根拠に）。
  - 不明点は**明示的な仮定**として成果物（機能仕様等）に書き切り、後から人間が根拠と前提を追えるようにする。
  - ただし**破壊的・不可逆操作の承認ゲート（FR-22）は維持**し、重大で不可逆な前提・判断は報告して可視化する（意思決定の移譲＝停止しないこと、と FR-22 ゲートは別物）。
  - 接続規約の詳細（対話前提スキルへの非対話委譲）は plugin の `${CLAUDE_PLUGIN_ROOT}/docs/architecture.md` §3.9.2（背景。委譲の要点は上記で自己完結）。
- **【承認ゲート FR-22】** 真に破壊的・不可逆な操作（**マージ〔main 統合〕／本番デプロイ・公開／削除〔データ・ブランチ強制削除・`gh repo delete`〕／本番影響／force-push・`reset --hard`・`clean -f` 等の履歴破壊**）は**実行せず**、**内部の提案アーティファクト**（patch / handoff ノート＝作業用クローンのローカルブランチ＋コミット）を残して報告する。**マージ・本番影響は人間承認後**に行う。
  - **topic ブランチへの push・PR（draft 含む）作成はサイクル内で自律実行してよい**（可逆な外向き操作。close / ブランチ削除で戻せる）。人間の判断ポイントは「PR の作成」ではなく **PR レビュー＆マージ**に集約する。
  - **運用注記**: 自律で PR を作成したら **PR URL を必ずサイクルレポートに出す**（可視性を担保）。CI・レビュアー通知コストがあるため、**意味のある単位にまとめて**乱発しない。段階ゲートを敷きたいポジションは **draft で自律作成し ready-for-review 化のみ人間**、という運用も選べる（`positions/<domain>.md` の権限節で指定）。

### 4. 検証 ── 対象: 実装完了分（ステータス → 検証中）
- テスト／レビュー／動作確認を接続ツールで実施する。レビューをサブエージェント／別セッションで行う場合は、前提知識（`tacit` 等）を**レビュー観点としてブリーフに明記**し、暗黙知違反を検出させる。
- **【承認ゲート FR-32】** 最終確認は人間。検証結果を提示して保留する。

### 5. 学習（agent-memory スキルの save に委譲）
- `experience` を記録（効いた手順・失敗・レビュー指摘の傾向）。
- **good/bad の記録（自己改善ループ向け・append のみ／改修しない）**: その周の good / bad を `experience` に `outcome`・`target`・`signal`（・任意で `recurrence`）付きで残す。ここでは評価の集計や skill/サブエージェントの改修はしない。傾向の集計と改修提案は別スキル **reflect**（低頻度）が担う（設計は `${CLAUDE_PLUGIN_ROOT}/docs/self-improvement.md`）。
- 新たに判明した暗黙知を `tacit`（`confidence` 付き）で記録する。
- 課題のステータスを更新する。

### 6. 報告・コミット
- サイクルレポートを出力（処理件数 / 保留中の承認 / 要相談 / 次アクション）。
- 変更（台帳・記憶）を Git コミットする（`--dry-run` 時はスキップ）。
- **秘密情報をコミット・ログに残さない。**

## 出力

- サイクルレポート（人間向けサマリ）。
- 更新された `challenge-ledger.md` / `memory/`。

## 注意

- 接続ツールが無い領域では「調査・計画・記憶更新」までを行い、実作業は保留として報告する。
- 横断課題のファンアウト（複数ドメイン並列）は、必要に応じてワークフローの型で実装してよい。
- すべての段階で承認ゲート（本文中の **【承認ゲート】**）を維持する。
- **ハーネス自体（skill/サブエージェント/ブリーフ）の改修は本サイクルで行わない**。step 5 で good/bad を記録するだけにして、改修は別スキル reflect（低頻度）に委ねる（コストを抑えるための分離）。
