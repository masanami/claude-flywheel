# run-cycle

Flywheel の 1 周（観測 → 整理 → 計画 → 実行 → 検証 → 学習 → 報告）を実行するスキル。
[自律実行ランタイム](../../templates/runtime/README.md)のスケジュール実行（routine/cron）から定期起動される「サイクル本体」。

> 設計: [architecture.md §5.2/§7/§3.8](../../docs/architecture.md)。状態は課題台帳・記憶・ポジション（すべてファイル）。
> **ツール非依存**: 実作業は各ポジションが指定する接続ツールへ委譲する（開発例: claude-harness）。本スキルは特定ツールに依存しない。

## 前提

- ブートストラップ済み（`positions/`・`memory/` が存在。未整備なら [bootstrap-domain-map](../bootstrap-domain-map/SKILL.md) を先に実行）。
- 利用先ワークスペースに `challenge-ledger.md`（課題台帳）が存在する（未整備なら [flywheel-init](../flywheel-init/SKILL.md) で生成）。

## 入力（任意）

- 対象ドメイン（省略時は全ドメイン）。
- `--dry-run`: 変更・委譲・コミットを行わず、「やること」だけ報告する。

## 冪等性の原則（cron で安全に回すため）

- 各課題の**ステータスに基づき、着手可能なものだけ**処理する。
- 何度起動しても**二重実行しない**（ステータスで判定）。
- 人間承認ゲートに達した項目は**「提案を残して保留」し、サイクルは止めない**（スケジュール実行では人間をインラインで待たない）。
- 1 周で扱う件数に上限を設けてよい（残りは次サイクル）。

## 手順

### 0. 観測
- `challenge-ledger.md` を読み込む。
- 関係する `positions/<domain>.md` と `memory/<domain>/INDEX.md` をロードする。

### 1. 整理（オーケストレーター）── 対象: ステータス「未分類」
- 各課題の影響範囲をドメイン地図（記憶 `map`）で推定する。
- 分類して「分類欄」を更新（[形式](../../docs/challenge-ledger-format.md)）:
  - **単一ドメイン** → 担当ドメインを割当、ステータス → 分類済。
  - **横断** → サブ課題へ分解し主担当＋協働を割当、ステータス → 分類済。
  - **要相談** → 人間キューへ（保留）。報告に列挙（承認ゲート #2）。

### 2. 計画（ドメインエージェント）── 対象: ステータス「分類済」で自ドメインに関係するもの（自己選択）
- 関連記憶を前ロードする。
- 目標 → タスクに分解し、タスク案を課題へ追記する。
- **【承認ゲート FR-13】** タスク起票は人間承認。「要承認」を付与して保留し報告。承認済みのみ次へ進む。

### 3. 実行 ── 対象: 承認済みタスク（ステータス → 着手中）
- ポジションが指定する**接続ツールへ委譲**して実作業する。
- **【承認ゲート FR-22】** 破壊的・不可逆操作（マージ／公開／削除／本番影響）は**実行せず**、提案（例: PR ドラフト）を残して報告する。

### 4. 検証 ── 対象: 実装完了分（ステータス → 検証中）
- テスト／レビュー／動作確認を接続ツールで実施する。
- **【承認ゲート FR-32】** 最終確認は人間。検証結果を提示して保留する。

### 5. 学習
- `experience` を記憶へ追記（効いた手順・失敗・レビュー指摘の傾向）。
- 新たに判明した暗黙知を `tacit` に `confidence` 付きで追記する。
- 課題のステータスを更新する。

### 6. 報告・コミット
- サイクルレポートを出力（処理件数 / 保留中の承認 / 要相談 / 次アクション）。
- 変更（台帳・記憶）を Git コミットする（`--dry-run` 時はスキップ）。
- **秘密情報をコミット・ログに残さない。**

## 出力

- サイクルレポート（人間向けサマリ）。
- 更新された `challenge-ledger.md` / `memory/`。

## 注意

- 接続ツールが無い領域では「調査・計画・記憶更新」までを行い、実作業は保留として報告する（ツール非依存）。
- 横断課題のファンアウト（複数ドメイン並列）は、必要に応じてワークフローの型で実装してよい。
- すべての段階で承認ゲート（[architecture.md §6](../../docs/architecture.md)）を維持する。
