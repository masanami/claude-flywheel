# start-day

その日の自走を始める「起動スキル」。board（または人間）が各エージェントワークスペースで `claude "/claude-flywheel:start-day"` を実行するだけで、cadence 設定の読み込み → 初回 `run-cycle` → セッション内 cron 登録までを行う。

> **セッション内 cron 方式を採用**: Claude Code のセッション内 cron（CronCreate 等）は **REPL アイドル時のみ発火**するため、`run-cycle` 実行中に次の定期便が来ても終了まで待つ＝セッション内で自然に直列化される（`.flywheel/cycle.lock` は手動併走への保険として引き続き機能）。**recurring は 7 日で自動失効**するが、日次で本スキルを起動する運用では到達しない。**対話割り込み中は発火が遅延**する（承認対話を邪魔しない望ましい挙動）。セッション（board の埋め込みターミナル等）を閉じれば cron ごと消滅する＝「board を閉じたら自走も止まる」安全なキルスイッチになる。詳細な採用理由・不採用選択肢（launchd／クラウド routine）は upstream の要件チケットおよび `docs/architecture.md` §7 を参照。

## 前提

- ブートストラップ済み（`positions/`・`memory/`・`challenge-ledger.md` が存在。無ければ `flywheel-init` / `bootstrap-domain-map` を先に実行）。
- Claude Code のセッション内 cron ツール（CronCreate 等）が使えるランタイムで、対話セッションとして起動していること（cron はセッションに紐づく）。

## 入力（任意）

- なし。cadence 設定はワークスペースの `.flywheel/cadence.json` から読む。

## 手順

### 1. cadence 設定の読み込み

- `.flywheel/cadence.json` を Read する（雛形は `${CLAUDE_PLUGIN_ROOT}/templates/cadence.json`。`flywheel-init` が scaffold 済みのはず）。
- **存在しない、または JSON としてパース不可**の場合は既定値で続行し、報告に「cadence 設定が無い（または壊れている）ため既定値で動作している」旨を明記する。既定値:
  - 業務曜日: 月〜金（cron の曜日フィールドで `1-5`。`0`/`7`＝日曜、`6`＝土曜という一般的な crontab 表記に従う）
  - 業務時間: `10:00`–`18:00`
  - run-cycle 間隔: 90 分
  - 発火分オフセット: **未指定（`null` を含む）が既定**。その場合は**ワークスペース名（ディレクトリ名）から決定的に導出**した値を使う（例: 名前の文字コード和を 59 で割った剰余 + 1。結果が `0`/`30` なら 1 ずらす）。導出した値は `.flywheel/cadence.json` の `cron_minute_offset` に**書き戻して固定**する（fleet 内の他エージェントと発火分が重なりにくく、かつ翌日以降も同じ値で安定させるため。テンプレートが全ワークスペースに同一の固定値を配ると、ずらす目的自体が無効になるので既定値は持たない）
  - reflect しきい値: 10 周ごと（`reflect.every_n_cycles`。同種 bad の再発しきい値は reflect スキル既定の `recurrence ≥ 2` をそのまま使い、cadence.json には持たせない＝二重定義を避ける）
- ファイルは存在するが一部フィールドのみ欠けている場合は、その項目だけ既定値で補い、欠落したフィールド名を報告に含める（全欠落と部分欠落を区別して報告する）。

### 2. 初回 run-cycle の即時実行

- `run-cycle` スキル（`${CLAUDE_PLUGIN_ROOT}/skills/run-cycle/SKILL.md`。Skill ツールがあれば `claude-flywheel:run-cycle` として呼び出してよい）を通常モード（`--dry-run` なし）で 1 周実行する。
- cron 登録前に必ず 1 回実行する（設定ミス・権限不備の早期検知、および「起動したら動き出す」体感のため）。
- 実行結果（サイクルレポート）はこのスキルの最終出力にそのまま含める。

### 3. run-cycle 定期便（recurring cron）の登録

- **登録の前に、既存の flywheel cron ジョブを掃除する（再実行の冪等化・必須）**: CronList 等で本手順・手順5の名前プレフィックスに一致する既存ジョブを検出し、残っていれば**全削除してから**登録する（同一ワークスペースでの本スキル再実行・前回分の残骸による二重登録を防ぐ。「何度起動しても二重実行しない」冪等原則を拍動レイヤーにも適用する。見つからなければ何もしない）。
- 手順1の cadence（業務開始・終了・run-cycle 間隔・発火分オフセット）から、当日の発火時刻の一覧を算出する（業務終了時刻**以降**の発火は含めない）。
  - 例: 開始 `10:00`・間隔 90 分・オフセット 5 分 → `10:05, 11:35, 13:05, 14:35, 16:05, 17:35`
- 算出した発火時刻を**分の値ごとにグルーピング**し、標準 cron の 5 フィールド（分 時 日 月 曜日・ローカルタイム）へ変換する。1 グループ＝1 recurring cron ジョブとして CronCreate で登録する（例: 上記なら `5 10,13,16 * * 1-5` と `35 11,14,17 * * 1-5` の 2 ジョブ。間隔が 60 分の約数で単一の時間ステップに収まる場合は 1 ジョブで足りる）。
  - **変換の自己検証（必須）**: 登録前に、変換した cron 式を発火時刻の一覧へ**逆展開**し、冒頭で算出した一覧と完全一致することを確認する（グルーピング・変換は決定的な機械処理であり、誤ると静かに誤時刻で発火するため。scripts/ への抽出は [#46](https://github.com/masanami/claude-flywheel/issues/46) の抽出基準〔散文でのレビュー往復・複数箇所での重複・静かに壊れる〕に照らして運用の実痛が出てから判断する）。
  - **cron 構文のフォールバック**: 環境の cron ツールが時フィールドのリスト（`10,13,16` 等）を受理しない場合は、**1 発火時刻＝1 ジョブ**（`分 時 * * 曜日` の固定形）に分割して登録する（ジョブ数は増えるが、常に受理される最小構文で確実に動かすため）。
- **分はエージェントごとにずらす**（fleet で複数エージェントを同時運用するため。`cadence.json` の発火分オフセットを使う。全員が `:00`/`:30` に発火すると API バーストとサブスク消費が重なるため避ける）。
- **ジョブ名は discoverable な共通プレフィックス**（例: `flywheel-run-cycle-<ワークスペース名>-<連番>`）を付けて登録する。複数グループに分かれる場合でも、後続の手順（手順4の兄弟ジョブ削除・手順5の締めジョブ）が CronList 等で**名前パターンで一括検出**でき、個々の ID を手渡しで追跡し続けなくてよいようにするため（1 グループ目の登録時点では 2 グループ目の ID がまだ確定していない、という手順4向けの相互参照問題を避ける）。
- 各ジョブのプロンプトには、`/claude-flywheel:run-cycle` の実行に加えて**手順4の課題枯渇時の自動停止判定**を含める。
- 登録した各ジョブの ID（および上記の名前プレフィックス）を控えておく（手順5の締めジョブ・手順4の兄弟ジョブ削除が対象を特定するため）。

### 4. 課題枯渇時の自動停止（各定期便のプロンプトに含める判定）

手順3で登録する run-cycle 定期便のプロンプトには、`run-cycle` 完了後に次を行うよう明記する（`run-cycle` スキル自体は変更しない。判定はこの定期便プロンプト側の指示として持たせる）:

- 台帳（`challenge-ledger.md`。アーカイブ `challenge-archive*.md` を含めて確認）を確認し、**未完了の課題が 1 件も無い**（全課題が完了・アーカイブ済み。かつ当周の観測ステップ〔ingest〕でも新規取り込みが無かった）場合:
  - この定期ジョブ（および同じく登録済みの他の run-cycle 定期便ジョブがあればそれも）を削除する。**他ジョブの削除は、手順3の名前プレフィックスで CronList 等から検出して行う**（このジョブが作成された時点で他ジョブの ID が確定していないケースがあるため、ID の事前埋め込みに依存しない）。
  - 「課題が枯渇した。次は要件定義（課題の起票）から」と報告して終了する。
  - 手順5の締めジョブは、対象が無くなるため合わせて削除してよい（同様に名前パターンで検出。削除できない場合は締めジョブ側の自己防御＝「定期便が既に無ければ何もしない」に委ねてよい）。
- **`計画承認待ち` / `完了確認待ち` の課題が残っている場合は止めない**（人間の承認が入れば次周から前進できるため）。この場合の周回は軽い観測のみで終わってよい。

### 5. 就業前の締めジョブ（one-shot cron）の登録

- 業務終了時刻の直前（例: 終業 15 分前。cadence に専用の項目が無ければこの固定値を使う）に発火する **one-shot** の cron を CronCreate で登録する。
- プロンプトには次を含める:
  1. 本日のサイクルサマリ（当日分の journal・台帳の状態）を報告する。
  2. 手順3で登録した run-cycle 定期便ジョブを削除する（すでに手順4の自動停止で自己削除済みなら何もしない）。
  3. **reflect しきい値判定**（判定根拠は `journal/index.jsonl`＝1 周 1 行の機械可読ジャーナル）: `index.jsonl` の総行数 `T` と当日分の行数 `t` を数え、`N = reflect.every_n_cycles`（既定 10）として **`floor(T / N) > floor((T − t) / N)`**（＝当日の周回で N の倍数を跨いだ）場合、または同種 bad の `recurrence ≥ 2`（reflect スキル既定）が観測される場合**のみ** `/claude-flywheel:reflect` を起動する。`index.jsonl` が無ければ倍数判定はスキップし、bad 再発条件のみ評価する。条件を満たさなければ reflect は起動せず、その旨を報告する（reflect は毎周回さない既存の起動タイミング規律を踏襲）。

### 6. 登録内容の報告

- 読み込んだ cadence 設定（既定値使用の有無・欠落フィールド）、初回 run-cycle の結果、登録した cron ジョブ一覧（run-cycle 定期便の cron 式・ID、締めジョブの発火時刻・ID）をまとめて報告する。

## 出力

- cadence 設定の読み込み結果（既定値使用の有無・欠落フィールドの明示）。
- 初回 `run-cycle` のサイクルレポート。
- 登録した cron ジョブ一覧（run-cycle 定期便・締めジョブそれぞれの cron 式または発火時刻・ID）。

## 注意

- **launchd（OS スケジューラ）／クラウド routine（`/schedule`）は使わない**（検討済みの不採用案。夜間自走が必要になったら再検討する。理由は upstream の要件チケットと `docs/architecture.md` §7 を参照）。
- 本スキルは**セッションが生存している間だけ**自走する。board（または対話セッション）を閉じると登録した cron ごと消滅する（安全なキルスイッチ。意図的な設計）。
- reflect の起動は締めジョブ経由のみに限定する（毎周は回さない。`reflect` スキルの起動タイミング規律と一致させる）。
- **ハーネス自体（skill/サブエージェント/ブリーフ）の改修は本スキルの範囲外**（`reflect` に委ねる）。
- 秘密情報を cron のプロンプト・報告に含めない（`run-cycle` 本体の原則を踏襲）。
- 同一ワークスペースでの再実行に対しては、**手順3冒頭の既存ジョブ掃除が必須**（best-effort ではない。掃除→再登録で冪等になる）。手順2の初回 `run-cycle` は再実行のたびに走るが、`run-cycle` 自体がステータス駆動で冪等・`.flywheel/cycle.lock` が並走を排他するため安全。
