# ドキュメント出力規約（AI 可読性）

> claude-flywheel のスキルが**生成するドキュメント**、および本リポジトリの docs を書くときの規約。
> 目的は **AI（後続のセッション・サブエージェント）が確実にパースし、文脈を取り違えないこと**。図や知識は人間だけでなく AI が読む前提で書く。
>
> この規約は docs に置くだけでは生成時に効かないため、**図/ドキュメントを生成するスキルは本ファイルを参照し、要点をスキル本文にもインライン明記する**（例: [bootstrap-domain-map](../skills/bootstrap-domain-map/SKILL.md) / [agent-memory](../skills/agent-memory/SKILL.md) / [reflect](../skills/reflect/SKILL.md)）。
> **本ファイルが唯一の正本**。スキル側のインラインは「**mermaid・言語タグ・キャプションの3点トリガ ＋ 本ファイルへのポインタ**」に留め、判断基準（ツリーは text 例外 等）は本ファイルだけが持つ（二重定義のドリフトを避ける）。
> **適用範囲**: claude-flywheel が生成する成果物（地図・ポジション・memory・内省レポート等）。run-cycle が委譲する独立 `claude -p` セッション（§3.9.2）が生成する成果物は**対象リポジトリの流儀に従う**ため本規約の射程外。エージェント自身の `memory/` への書き戻しは agent-memory スキル経由で本規約が効く。

## 1. 図は mermaid で描く（ASCII art を使わない）

- フロー図・関係図・状態遷移・シーケンスは **mermaid** コードブロック（` ```mermaid `）で描く。ASCII art（罫線 `│┌┐└─`、矢印 `▼▲→↓` を並べた擬似図）は使わない。
  - 理由: ASCII art は桁揃えがズレやすく、AI が構造（ノード・辺・向き）を復元しづらい。mermaid は構造がテキストで明示され、パースが安定し、GitHub でも描画される。
- **ディレクトリツリーは例外**: `tree` 構造は ` ```text ` のコードブロックのまま書く（mermaid 化はかえって読みにくい）。
- mermaid ラベル内で `<` `>` を使うときは `&lt;` `&gt;` にエスケープする（例: `memory/&lt;domain&gt;/`）。改行は `<br/>`。

## 2. 図には1行キャプションを併記する

- 各図（mermaid ブロック）の**直前**に、その図が何を表すかを1行のイタリックで書く（例: `*図: 2ループ — …*`）。
- 図を描画しない経路（プレーンテキストで読む AI）でも要点が伝わり、文脈把握が安定する。

## 3. コードフェンスに言語タグを付ける

- すべての code fence に言語を指定する（` ```mermaid ` / ` ```bash ` / ` ```yaml ` / ` ```text ` / ` ```markdown ` 等）。
- 無指定の ` ``` ` は使わない。タグがあると AI・lint（MD040）双方のパースが安定する。
  - ディレクトリツリー・擬似リスト・プレーン例 → ` ```text `
  - 実行コマンド → ` ```bash `

## 4. 安定 ID と相互参照

- 要件・未決事項・課題には**安定 ID**を振る（`FR-xx` / `NFR-xx` / `AO-xx` / `C-xxx`）。本文からは ID と節番号（`§3.9.2`）で参照する。
- ある決定が別ドキュメントに影響するときは**双方向に参照**を張る（リンク＋節番号）。「未確定」「#NN で扱う」のような表現は、確定したら速やかに確定先へ更新する（陳腐化させない）。

## 5. 構造化を優先

- 列挙・対応関係は**表**で書く。手順は番号付きリスト。図にすべきは「関係・流れ」だけにする。
- 1 ファイル＝1 トピック、見出しで階層化し、冒頭に1〜2行の要約を置く。
