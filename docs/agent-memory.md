# エージェントmemory 運用方針

> ドメイン担当エージェントが、**ドメイン地図だけでなく暗黙知・経験**を蓄積し、横断的に自律して動けるようにするための記憶の運用を定義する。
> ブートストラップ（[bootstrap-domain-map](../skills/bootstrap-domain-map/SKILL.md)）で初期 seed を生成し、業務遂行のたびに更新していく。
>
> - ステータス: ドラフト（検討中）
> - 関連: [requirements.md FR-40〜42](requirements.md)（学習フェーズ）, [challenges.md 課題2](challenges.md)

## 1. なぜ memory か

- 課題2の本質は「**人間が把握しきれていない業務フロー・サービス関係を、エージェントが知識と経験として持つ**」こと。
- コードに**書かれていない**暗黙知（業務ルール・命名の意図・落とし穴）や、**過去の試行から得た経験**（何が効いた/失敗した）は、毎回ゼロから読み解くとコストが高い。
- → これらを永続化し、レビュー・実装の前提知識として供給する。これは**課題1ボトルネックB（前提知識の不足）**の解にも直結する。

## 2. 記憶の種類（type）

| type | 内容 | 例 | 主な生成元 |
| --- | --- | --- | --- |
| `map` | ドメイン地図 | サービス一覧、サービス間呼び出し/イベント、ドメイン境界 | ブートストラップ・コード解析 |
| `tacit` | 暗黙知 | 業務ルール、命名の意図、設計上の制約、注意点 | コメント/README/人間からの確認 |
| `experience` | 経験 | 過去タスクの結果、効いた手順・失敗、レビュー指摘の傾向 | 実行・検証後の学習 |
| `reference` | 参照 | API 契約、スキーマ、外部ドキュメント・ダッシュボードの場所 | ブートストラップ・運用 |

## 3. 置き場所・構造

- **ドメインごとに記憶ストアを分ける**（横断知識は関係先ドメインへ相互リンク）。

```
memory/
  <domain>/
    INDEX.md                  # そのドメインの記憶インデックス（1行/件）
    map-<slug>.md             # ドメイン地図の断片
    tacit-<slug>.md           # 暗黙知1件
    experience-<slug>.md      # 経験1件
    reference-<slug>.md       # 参照1件
```

> 構造は既存ハーネスのファイルベース memory（frontmatter + INDEX）の規約に揃える。配置場所の最終決定は [requirements.md OQ-01](requirements.md) と合わせて確定する。

### ファイル frontmatter（案）

```markdown
---
name: <short-kebab-case-slug>
description: <one-line。recall 時の関連判定に使う>
domain: <ドメイン名>
metadata:
  type: map | tacit | experience | reference
  sources: [<根拠となるファイル/URL/会話>]
  confidence: high | medium | low   # 推定か確証かを区別（特に tacit）
---

<事実・知見の本文。関連記憶は [[name]] でリンク>
```

- 1 ファイル = 1 事実（粒度を小さく保ち、recall と更新を容易に）。
- `confidence` で「コードから確証」「人間に未確認の推定」を区別し、暗黙知の誤りを管理する。

## 4. ライフサイクル

```
ブートストラップ            実行・検証               レビュー前
  │ map/reference を seed     │ 学習: experience 追記   │ 関連 memory を事前ロード
  │ tacit 候補を抽出          │ 新発見の tacit を追記   │ （前提知識として供給）
  ▼                          ▼                        ▼
 初期記憶 ───────────────▶ 継続的に蓄積・更新 ──────▶ 質の高いレビュー/実装
```

- **生成**: ブートストラップで `map` / `reference` を seed、`tacit` 候補を抽出（人間確認は承認ポイント）。
- **更新（学習）**: タスクの実行・検証後に `experience` を追記（FR-40/41）。新たに判明した暗黙知も追記。
- **活用**: 要件・設計レビューや実装の**前ロード知識**として参照（課題1ボトルネックBの解）。
- **保守**: 人間が閲覧・修正・削除できる（FR-42）。`confidence: low` は要検証として扱う。

## 5. 検討中の論点（Open Questions）

- 配置場所: リポジトリ内（Git 履歴で追える）か、別ストアか（[OQ-01](requirements.md)）。
- 横断知識の共有: 複数ドメインで共有する事実をどう持つ（共有領域 or 相互リンク）。
- 暗黙知の検証フロー: `confidence: low` を誰が・いつ確証へ昇格させるか。
- recall の仕組み: タスク着手時にどの記憶を自動ロードするかの選定方法。
